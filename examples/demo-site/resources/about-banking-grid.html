Banking Grid with Natural-Language Integration

This example shows rows of loan data. The user can type natural language queries in the grid toolbar. The app connects to a backend NL model to interpret the user intent, and the backend performs actual filtering on the client-side store. It is also possible to navigate through pages using commands.

Usage

By using the search bar, you can type in natural language queries for filtering or sorting data. For example:

- Show personal loans
- Show interest rates between 5 and 5.5
- Show overdue accounts
- Sort by borrower a to z
- Sort by principal amount ascending
- Show educational and auto loans, sort by remaining balance descending
- Group by Status
- Go to last page

Files

Frontend (Ext JS)

  • /app/view/ai/DataGrid.js
    Banking Grid view: columns, toolbar with NL input and standard column filters, binds to the store.
  • /app/view/ai/DataGridController.js
    Captures NL input, sends it to the backend endpoint, receives the model's interpretation (e.g., a filter spec or structured query), and while it applies filtering locally to the store using store.filter(), the backend performs the actual filtering.
  • /app/store/BankingDataStore.js
    Client-side store that load sales rows from the backend. The store supports local filtering and sorting; data is loaded from the server after every user query.

Backend (Node.js)

  • /server/server.js
    Express bootstrap: serves static assets and exposes the NL endpoint used by the grid controller.
  • /server/endpoints/banking-endpoint.js
    Builds the NL prompt from the user text, calls the NL model API, and returns the model's structured interpretation (not raw rows). Note: because filtering runs a second step after the client-side store requests the filtered data, the NL backend returns a predictable filter description or JSON structure that the controller can consume.
  • /server/endpoints/banking-data.js
    Backend endpoint that retreives the filtered data.

Typical runtime flow (step-by-step)

  1. User types a free-text query in the grid toolbar.
  2. DataGridCcontroller.js sends the text to the backend NL endpoint (POST /api/ai-endpoint).
  3. banking-endpoint.js composes a hand-written prompt and calls the NL model. The model returns a structured interpretation.
  4. The controller receives the response and applies filters to the filters plugin.
  5. The plugin calls the server perfoming server-side filtering.

The response

Once the middleware responds, it returns a JSON object that can include up to four main sections: filters, sorters, grouping, and tokenData.

  • filters
    Contains the filtering conditions generated from the user's query, defining which records should be displayed. Ready to be applied to the grid filters.
  • sorters
    Specifies how the data should be ordered (e.g., by field and direction such as ascending or descending) ready to be applied to the store.
  • grouping
    Describes how records are grouped by a specific field.
  • tokenData
    Provides the total tokens consumed during the request.
