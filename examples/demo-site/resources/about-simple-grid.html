Sales Grid with Natural-Language Integration

This example shows rows of sales representative data. The user can type natural language queries in the grid toolbar. The app connects to a backend NL model to interpret the user intent, and the frontend performs actual filtering on the client-side store.

Usage

By using the search bar, you can type in natural language queries for filtering or sorting data. For example:

- Show active users under 30
- Show users between 40 and 50 years old
- Show active users with a live.com account
- Show USA users with most sales
- Sort by name a to z
- Sort by age ascending
- Show users from the UK with sales over 800, sort by age descending
- Group by type

Files

Frontend (Ext JS)

  • /app/view/ai/DataGrid.js
    Sales Grid view: columns, toolbar with NL input and standard column filters, binds to the store.
  • /app/view/ai/DataGridController.js
    Captures NL input, sends it to the backend endpoint, receives the model's interpretation (e.g., a filter spec or structured query), and applies filtering locally to the store using store.filter().
  • /app/store/SimpleDataStore.js
    Client-side store that holds sales rows. The store supports local filtering and sorting; data is either preloaded or fetched once from the server and kept client-side.

Backend (Node.js)

  • /server/server.js
    Express bootstrap: serves static assets and exposes the NL endpoint used by the grid controller.
  • /server/endpoints/simplegrid-endpoint.js
    Builds the NL prompt from the user text, calls the NL model API, and returns the model's structured interpretation (not raw rows). Note: because filtering runs in the frontend, the backend returns a predictable filter description or JSON structure that the controller can consume.

Typical runtime flow (step-by-step)

  1. User types a free-text query in the grid toolbar.
  2. DataGridCcontroller.js sends the text to the backend NL endpoint (POST /api/ai-endpoint).
  3. simplegrid-endpoint.js composes a hand-written prompt and calls the NL model. The model returns a structured interpretation.
  4. The controller receives the response and applies filters locally to the filters plugin. No server-side filtering is performed in this example.

The response

Once the middleware responds, it returns a JSON object that can include up to four main sections: filters, sorters, grouping, and tokenData.

  • filters
    Contains the filtering conditions generated from the user's query, defining which records should be displayed. Ready to be applied to the grid filters.
  • sorters
    Specifies how the data should be ordered (e.g., by field and direction such as ascending or descending) ready to be applied to the store.
  • grouping
    Describes how records are grouped by a specific field.
  • tokenData
    Provides the total tokens consumed during the request.
